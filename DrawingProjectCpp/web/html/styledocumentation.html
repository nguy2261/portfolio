Coding Style and Documentation

Coding Style

When writing a program of any size, it is very important to keep with a common style. This will not only make it easier for you, but also for those developing for the project in the future. One styling point that is particularly important is spacing. Take the following line of code for example:


if (imageFile.find(".jpg", imageFile.size() - 5) != std::string::npos)

You could easily write something like this as follows: 


if(imageFile.find(".jpg",imageFile.size()-5)!=std::string::npos)

Notice how the second example is a bit harder to read and looks very scrunched together. By using spacing where appropriate, you can make your code much easier to read and work with. This is just one of many important aspects of coding style. We can also see our bracket placement convention in the following piece of code. Notice how the are both directly lined up with the 'i' in 'if'. This is a very common way to place your statement and function brackets. 


        //Check for valid .jpg extension
        if (imageFile.find(".jpg", imageFile.size() - 5) != std::string::npos)
        {
                //We have a jpg image, so call the appropriate jpg load function
                cout << "Image extension is .jpg" << endl;
                image = JPGImage::load(imageFile);
        }

One could also write the statement as follows:


        //Check for valid.jpg extension
        if (imageFile.find(".jpg", imageFile.size() - 5) != std::string::npos) {
                //We have a jpg image, so call the appropriate jpg load function
                cout << "Image extension is .jpg" << endl;
                image = JPGImage::load(imageFile);
        }

This is also a very common way to place your brackets, but we have chosen to go the route of that in the first example. We have also implemented a naming a few naming conventions for particular classes. Take tools for example. Each tool file starts with 'T'. The same goes for filters, so blur tool files would be named 'TBlur.*', and blur tool files would be named 'FBlur.*'. This really helps to organize the code nicely in a directory. With this naming convention in place, all tools will be grouped together, as well as filters.


Documentation

Another important aspect of style is commenting your code. Comments help to better explain what is going on in your code. It is important to leave useful comments in your code where you feel necessary, but be wary of overdoing it. Too many comments can clutter the code. Notice that we have clearly indicated what the piece of code is doing with //Check for valid .jpg extension.

We then confirm the condition in which we execute the following statements with //We have a jpg image, so call the appropriate jpg load function. This piece of code is relatively self explanatory, and does not need any additional comments.

Please be sure to follow our styling and commenting conventions when adding your work to our code.
